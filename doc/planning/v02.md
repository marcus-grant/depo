# Version 0.2 Planning

Post-MVP improvements. Nothing here ships until MVP (v0.1) is complete.

## Schema change readiness (READ BEFORE ANY SCHEMA CHANGE)

The migration guard shipped in `ft/migrations`: `repo_meta` holds
`schema_version`, stamped on init and read back, and server-starting
commands refuse to run when the DB is behind or ahead of the code's
`SCHEMA_VERSION` (both directions raise `SchemaVersionError`). What did
NOT ship is the migrate command that applies pending files. A stale DB is
detected and blocked but cannot be brought forward except by hand.

The first task that changes the schema must, in the same PR:

- Build the migrate command that applies pending `migration-M-m-p.sql`
  files, forward-only, triggered explicitly (never at startup). The
  selection helpers already exist (`pending_migrations`, `list_migrations`,
  `_migration_version` in `repo/sqlite.py`); wire a command over them.
- Refactor migration code out of `repo/sqlite.py` into its own module
  under `repo` (for example `repo/migrations.py`). It currently sits
  beside unrelated row-mapping and repo code; the migrate PR is where it
  earns its own module.
- Add the upgrade test deferred from MVP: a DB at version N upgrades to
  N+1 without data loss.
- Document the migration system in `doc/module/repo.md`: the shipped guard
  and the new command.

Bump `SCHEMA_VERSION` in `repo/schema/__init__.py` and the stamp in
`_schema.sql` together; they are a deliberate double-entry (noted in the
schema file). A mismatch between them is what the guard exists to catch.

## Honest status codes on htmx error surfaces

`htmx_error` hardcodes `status_code=200` so htmx will swap the error partial
into the DOM. htmx only swaps 2xx responses by default, and the 200 convention
grew up as a workaround for that.

The cost is that an auth failure, a validation failure, and a success are
indistinguishable on the wire. Monitoring, logging middleware, and any API
client hitting the same endpoint all see 200.

htmx exposes `htmx:beforeSwap`, where a handler can inspect
`evt.detail.xhr.status` and set `shouldSwap = true`, so a real 4xx can carry a
partial and still render. Honest status and rendered partial are not in
conflict.

- [ ] Add a client-side `htmx:beforeSwap` handler opting into swapping 4xx
      bodies
- [ ] Return real statuses from `htmx_error` rather than 200
- [ ] Two specs flip and must be updated:
      `TestAuthRequiredHandler::test_htmx_surface_returns_200_partial` and
      `TestUploadGate::test_htmx_rejection_carries_login_link`
- [ ] Update `doc/design/routes.md`, which documents the 200 as a temporary
      exception to the uniform-401 rule

This also answers the open question in the login form chrome block: htmx *can*
swap an error partial while preserving a 401, but only with a `beforeSwap`
handler. Plan the login error response on top of this.

## Global Chrome & Layout Realignment (nav / main / footer / base.html)

**Goal:** Make the overall page shell feel like a single calm system surface.
Prefer structural honesty (borders, spacing, rhythm) over “window/dialog” cosplay.
Ensure all hierarchy works in pure grayscale; color remains semantic only.

### Styling Refinement Pass (deferred from ft/visual-refinement)

- Merge info header row
- Fix View Raw button height
- Resolve upload page width (too narrow despite `main--wide`)
- Replace `article` with `section` on info page

### Interaction semantics stay semantic

- Keep warm/cool meaning invariant across the shell:
  - cool = focus/state
  - warm = attention/interruption
  - red = failure only
- Do not use accent color to “decorate” global chrome.

### Primitive audit (enforce allowed tools)

- Allowed:
  - spacing,
  - typography (mono for identifiers),
  - weight,
  - sizing steps,
  - hard borders,
  - optional dither separators.
- Disallowed by default:
  - blur shadows, gradients, large colored surfaces, decorative textures.
- If a new visual device is introduced
  - it must justify itself as structure (not vibe).

### HTMX compatibility (no layout surprises)

- Shell (`header/main/footer`) should remain stable across HTMX swaps.
- Swaps should target interior regions only;
  - avoid global reflow by keeping consistent container widths/padding in `base.html`.

### Login form chrome (deferred from ft/login-session)

- Style the auth/login.html form and error elements (login__form,
  login__error); both are currently undecorated structural markup
- Add coverage for the login__form and login__error classes in the
  template class-consistency / spec test modules
- Add required and rely on type="email" on the login inputs for
  client-side validation
- Inform the user of a bad login via a partial update rather than a
  full-page re-render. HTMX is the candidate, but verify first whether
  it can swap an error partial while preserving the 401 status, or
  whether it only swaps on 2xx by default and needs configuration for
  error codes. It may not be the right mechanism; find out before
  committing
- Based on that finding, plan the idiomatic login error response
  (status code, swap target, partial template)
- When this lands, update the affected tests: the BS4 form-structure
  checks and error-block assertions in test_routes.py assume a
  full-page render and will change

## Early v0.2 housecleaning

- Improve `get_templates()` ergonomics: the call-it-everywhere pattern
  is awkward; build a cleaner shared singleton for the templates
  instance that call sites can reference directly
- Default `follow_redirects=False` in the test client fixtures; `TestClient`
  defaults it to True but route tests rarely want redirects followed, so nearly
  every redirect assertion has to override it
- Reconsider whether the error classes belong in the model layer rather
  than util; they increasingly carry structured domain fields (id,
  resource, email) rather than being pure utilities
- Lift the per-instance severity override into DepoError.**init** so all
  subclasses inherit it the way they inherit the status override, rather
  than each class reimplementing it
- Split the single errors module into per-domain modules once it grows
  large enough that one file hurts maintenance
- Rename `tests/factories/db.py::insert_user` to `seed_user` to resolve
  naming overlap with `SqliteRepository.insert_user`
- Split `src/depo/repo/sqlite.py` into per-concern submodules (items,
  users, schema) once the auth sequence lands
- Standardize scrypt cost params in password tests to minimum values
  (n=2, r=1, p=1); some tests still use n=2**14 making the suite slow
- Revisit scrypt N=2**16 choice against a timing benchmark on target
  hardware; OWASP floor is 2**17, current value is a split-the-difference
  estimate without a measured baseline
- Replace empirical maxmem formula (256*n*r*p + 1MiB) in password.py
  with a principled bound once OpenSSL's internal buffer requirement is
  confirmed
- Add redirect-when-unauthenticated to GET /logout (mirroring the
  authenticated-redirect already on GET /login), once more of the app
  exists to redirect to sensibly
- Collapse the client fixture matrix. Request flavor (browser, htmx) is a
  header concern, not a client identity: keep an unauthenticated and an
  authenticated client, apply `HEADER_BROWSER`/`HEADER_HTMX` per request, and
  retire `t_browser` and `t_htmx`. Rename `t_client` to `t_guest` so every call
  site states its auth state. Split the client fixtures into their own submodule
  while the call sites are already being touched
- Add typed test accessors (`app_repo(client)`, `app_store(client)`) to replace
  the `cast(FastAPI, client.app).state.X` chains. Starlette's `State` is
  dynamically attributed, so those chains defeat LSP navigation entirely. Sweep
  with `grep -rn "\.state\b" tests/`
- Annotate fixture return types at the test-param boundary. Pyright cannot
  connect a test's fixture parameter to the fixture function, so anything
  reached through a fixture is typechecked vacuously. This hid broken
  `orch.ingest` calls until the `OrchEnv` alias was added and pyright could
  finally see them
- Tidy the `tests/factories` module: imports and groupings have accreted, and
  finding an existing factory is harder than it should be
- Make `repo.insert`'s `uid` and `perm` required, as `ingest` now is. The
  defaults are currently unreachable from any live path, but they are the same
  silent-anonymous-write hazard one layer down
- Give `ingest` a structured input object rather than seven loose kwargs.
  Callers splat a TypedDict into it with a `# type: ignore` at every call site.
  The web layer's `UploadParams` union is the right shape but the wrong name for
  the service boundary
- Resolve the authenticated-probe-client verbosity in `tests/web/test_deps.py`:
  testing an authenticated request against a probe route currently spins up one
  app to mint a signed session cookie and a second to consume it
- Decide who owns a deduped item. A second user uploading identical bytes
  currently receives the first user's item, carrying the first user's uid.
  Inherent to content-addressed dedupe, but the ownership and visibility
  semantics are unstated
